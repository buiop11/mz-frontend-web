# 페이지 구조 가이드

Next.js App Router 기반 프로젝트에서 페이지 구성 방법에 대한 가이드입니다.

## 체크리스트(새 페이지 생성 시 확인사항)

- SEO가 필요한가?
- 서버 데이터 패칭이 필요한가?
- 다른 페이지와 공통 레이아웃이 있는가?

## 1. SEO가 필요 없는 페이지 (ex) Form 페이지)

**특징:**

- 사용자 입력 폼이 주된 기능
- 검색 엔진 최적화가 불필요
- 클라이언트 사이드 렌더링으로 충분

**구조:**

```
src/app/(app)/support/qna/
└── page.jsx  ← 모든 로직과 UI를 하나의 파일에 작성
```

---

## 2. 서버 데이터 기반 렌더링이 필요한 페이지

**특징:**

- 서버에서 데이터를 가져와 렌더링
- SEO 최적화 필요

**구조:**

```
src/app/(app)/growth/market/
├── page.jsx   ← 데이터 패칭 로직 (Server Component)
└── comp.jsx   ← UI 컴포넌트 (Client Component)
```

**역할 분담:**

- `page.jsx`: 서버에서 데이터 패칭, 메타데이터 설정
- `comp.jsx`: 실제 UI 렌더링, 클라이언트 상호작용

**예시:**

- 상품 카탈로그 페이지
- 검색 결과 페이지

---

## 3. 페이지별 레이아웃 처리

### 3.1 페이지 통합 레이아웃

- 해당 페이지에서만 사용하는 레이아웃
- page.jsx에 레이아웃과 페이지 로직을 함께 작성

### 3.2 별도 레이아웃 파일 (특별한 경우에만)

**언제 사용:**

- 여러 페이지에서 공통으로 사용하는 레이아웃

**구조:**

```
src/app/(app)/faq/
├── layout.jsx  ← 공통 레이아웃 (키비주얼)
├── (page)
│   └── page.jsx
└── [id]/
    └── page.jsx
```

---

## 4. 라우트 그룹 네이밍 규칙

**규칙:** 페이지로써의 의미상 감싼 부분은 모두 `(page)`로 통일

**예시:**

```
src/app/(app)/
├── support/
│   ├── notice/
│   │   └── (page)/   ← 목록 페이지
│   └── faq/
│       └── (page)/   ← 목록 페이지
```
